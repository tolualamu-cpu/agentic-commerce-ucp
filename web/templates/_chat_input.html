{# Chat input form. Sits BELOW the conversation log in document flow. Default behaviour: POST /chat (HTMX) and let the SSE handler stream the reply into the visible #chat-log — no navigation. Used by the active /chat state. When included with ``reveal_in_place=True`` (empty /chat, where the chat-log starts hidden), the form does NOT navigate. Instead it calls ``window.__chatRevealActive()`` on a successful POST, which un-hides #chat-log and drops the hero so the SSE-streamed reply lands in the now-visible log. CRITICAL: this keeps the page's SINGLE, already-open EventSource('/chat/stream') connection (owned by _toast.html) alive throughout. A full reload (the old ``reload_after`` behaviour) tore that connection down and re-opened a new one; during the unload gap the dying connection could still drain the orchestrator's products/text/done burst off the single-consumer sse_queue — events that are NOT re-queued — so the freshly-loaded page got nothing and the chat "didn't populate until you flipped to another page and back" (history re-render). No reload ⇒ no second connection ⇒ no race. ``reload_after=True`` is retained ONLY for genuine cross-page hops (e.g. the home hero posting then navigating to /chat) where there is no in-place log to reveal. The form id is `chat-form` and the input id is `chat-input` — same names the SSE handler and suggestion-chip helper expect.
